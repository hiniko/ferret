# Setup & integration modes

Ferret meets you at one of three levels of involvement. Pick the row that matches how much
you want it to manage.

| Mode | Session | Schema (indexes / sidecars / triggers) | Reindex |
| --- | --- | --- | --- |
| **EF Core** (batteries-included) | `Ferret.EntityFrameworkCore` | Generated for you by `Ferret.Migrations` via `dotnet ef migrations` | Inline at migration time, or the reindex worker / CLI |
| **Dapper** | `Ferret.Hydration.Dapper` | Generated by `Ferret.Migrations` (if you keep an EF `DbContext` for migrations) **or** hand-rolled SQL | `engine.ReindexAsync<T>` or `dotnet ferret reindex` |
| **Take all responsibility** (BYO) | Implement `IFerretSession` | You write all the DDL yourself | You manage it |

All three run the **same engine and SQL** — the only differences are who opens connections,
who creates the schema, and who triggers reindexing.

---

## EF Core — batteries-included

Ferret manages hydration, schema, and reindexing.

```csharp
services.AddDbContext<AppDbContext>(o => o.UseNpgsql(connectionString));
services.AddFerret(opts => opts
    .ScanAssembly(typeof(Product).Assembly)
    .UsePostgres()
    .UseTrigramSearch());          // + UseFullTextSearch / UseVectorSearch / UseHybridSearch
```

Schema is emitted from your `[Searchable]` attributes — see
[migrations.md](migrations.md). Query via the `DbContext` extension:

```csharp
var page = await dbContext.SearchOffsetAsync<Product, Guid>(serviceProvider, query);
```

## Dapper — you own the connection, Ferret does the rest

Use this when you are not on EF Core for queries. You supply a `DbConnection` factory; Ferret
hydrates with Dapper.

```csharp
services.AddFerret(opts => opts
    .ScanAssembly(typeof(Product).Assembly)
    .UsePostgres()
    .UseTrigramSearch()
    .UseDapperHydration());

await using var session = new DapperSession(
    ct => Task.FromResult<DbConnection>(new NpgsqlConnection(connectionString)),
    sp.GetRequiredService<ISqlDialect>());

var page = await sp.GetRequiredService<IFerretEngine>()
    .SearchOffsetAsync<Product, Guid>(session, query);
```

For schema you still have two choices: keep a thin EF `DbContext` purely to run
`Ferret.Migrations` ([migrations.md](migrations.md)), or hand-roll the DDL
([schema-reference](migrations.md#manual-schema-reference)). Reindex with
`engine.ReindexAsync<T>(session, "<group>")` or the `dotnet ferret reindex` CLI.

## Take all responsibility — bring your own session

For any other data stack, implement `IFerretSession` (open a connection, expose the
`ISqlDialect`) and Ferret will build and run its SQL through it. You own every piece of
schema (extensions, indexes, sidecar tables, sync triggers) and you own reindexing. Ferret
contributes only the query construction and result shaping.

This is the escape hatch — everything above is built on the same `IFerretSession` seam.
